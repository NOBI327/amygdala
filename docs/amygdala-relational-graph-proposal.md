# Amygdala 機能追加提案書：感情タグ付き関係性グラフ（Emotion-Tagged Relational Graph）

## 1. 概要

Amygdalaに「感情タグ付き関係性グラフ」を追加する。会話に登場する人物・話題・アイテム間の関係性を網（グラフ）構造で保持し、各エッジに感情ベクトルとタグを付与する。タグとエッジは時間経過で減衰し、関係性の変化を自然に表現する。

本機能はAmygdalaの既存アーキテクチャ（Frontman/Backman、10軸感情ベクトル、soft-archive、event-driven search）を前提とし、どのフェーズからでも独立して組み込める純粋な機能追加として設計する。

---

## 2. 背景と動機

### 2.1 現状の課題

Amygdalaの現行メモリシステムは「個別の記憶」を感情ベクトル付きで保持している。しかし、記憶間の関係性は明示的に管理されていない。

具体例：

- ユーザーが「上司」について複数回言及している場合、個別の記憶としてはそれぞれ保持されるが、「上司との関係性がどう変遷しているか」はメモリから読み取れない
- 「プロジェクトA」と「技術B」が頻繁に共起しても、その結びつきの強さは記録されない
- ユーザーが過去に不快だった話題に再度触れたとき、その文脈を自然に考慮した応答ができない

### 2.2 公式機能との差別化

Anthropic公式のmemory機能（Claude Code auto memory、claude.ai memory）は事実の永続化に特化している。「ユーザーはPythonを使う」「ユーザーの上司はAさん」といった事実的知識の管理。

本提案は、事実の背後にある「感情的関係性」と「関係性の時間変化」を捕捉する。これは公式のロードマップに現時点で存在しない領域であり、Amygdalaの独自ポジションを強化する。

## 3. 設計

### 3.1 グラフ構造

#### ノード（Node）

会話に登場するエンティティ。

```
Node {
  id: string                    # 一意識別子
  type: enum                    # person | topic | item | place | event
  label: string                 # 表示名（例："上司", "Amygdalaプロジェクト"）
  first_seen: datetime          # 初出日時
  last_seen: datetime           # 最終言及日時
  mention_count: int            # 累計言及回数
  base_emotion: EmotionVector   # このエンティティに対する基底感情
}
```

type の分類：

| type | 定義 | 例 |
|------|------|-----|
| person | 人物 | 上司、同僚、家族、友人 |
| topic | 話題・テーマ | 仕事、転職、AI開発 |
| item | 具体的なもの・ツール | Python、Notion、特定のプロジェクト名 |
| place | 場所 | 職場、自宅、特定の都市 |
| event | 出来事 | 面接、発表、トラブル |

#### エッジ（Edge）

ノード間の関係性。本提案の中核。

```
Edge {
  source: NodeID
  target: NodeID
  tags: List[Tag]               # 関係性の性質を表すタグ群
  emotion_vector: EmotionVector # Plutchik 8軸 + importance + urgency
  strength: float (0.0-1.0)    # 接続強度（減衰対象）
  confidence: float (0.0-1.0)  # 推論信頼度（サンプル数ベース）
  last_activated: datetime      # 最終活性化日時
  activation_count: int         # 累計活性化回数
}
```

#### タグ（Tag）

エッジに付与される関係性の性質ラベル。タグ自体も減衰する。

```
Tag {
  label: string                 # タグ名（例："ストレス源", "協力者", "学習対象"）
  created: datetime             # 生成日時
  strength: float (0.0-1.0)    # タグ強度（減衰対象）
  activation_count: int         # このタグが活性化された回数
  decay_rate: float             # タグ固有の減衰率
}
```

### 3.2 タグの減衰設計

#### 減衰率のカテゴリ

タグの性質に応じて減衰率を3段階に分類する。

| カテゴリ | decay_rate | 対象例 | 根拠 |
|----------|-----------|--------|------|
| 安定関係 | 0.01/day | 家族、親友、信頼できる同僚 | 人間関係の基盤は容易に変化しない |
| 状況依存 | 0.05/day | ストレス源、協力者、競合相手 | 状況の変化で関係性が変わりうる |
| 一時的 | 0.10/day | 特定プロジェクトの担当、今週の課題 | 短期的な文脈に紐づく |

#### 減衰の計算

```
current_strength = initial_strength × e^(-decay_rate × days_since_last_activation)
```

- 再活性化（会話で言及されたとき）で strength がリセットされる
- strength が閾値（例：0.1）を下回ったタグはエッジから除去される
- エッジ上の全タグが除去された場合、エッジ自体が soft-archive 対象となる

#### タグの生成と昇格

タグは以下のパイプラインで生成・管理される。

```
会話から関係性を検出
  → 候補タグを生成（confidence: low）
  → 同一タグが再度検出される（activation_count++）
  → activation_count ≧ 閾値（例：3回）で正式タグに昇格（confidence: high）
```

この「候補→昇格」パイプラインにより、一度の会話から過剰な推論を行うリスクを軽減する。

### 3.3 グラフのスコープ管理

#### 肥大化防止

| 制約 | 値 | 根拠 |
|------|-----|------|
| アクティブノード上限 | 100 | Frontmanのコンテキストバジェットを圧迫しない範囲 |
| ノードあたりエッジ上限 | 20 | 検索時の情報爆発を防止 |
| エッジあたりタグ上限 | 10 | タグの意味的重複を防止 |

上限を超えた場合、strength × confidence の積が最も低いものから soft-archive する。

#### 検索時のホップ制限

Backmanがグラフを探索する際のルール：

- デフォルト：1ホップ（直接接続のみ）
- 明示的に関連が必要な場合：最大2ホップ
- 3ホップ以上は禁止（情報爆発のリスクが利益を上回る）

### 3.4 既存アーキテクチャとの統合

#### Backmanとの統合

グラフの管理はBackmanの責務に追加する。

```
Backman（既存）
  ├── 感情ベクトル算出
  ├── メモリ検索（event-driven search）
  └── soft-archive管理

Backman（追加）
  └── 関係性グラフ管理
       ├── ノード/エッジの生成・更新
       ├── タグの生成・昇格・減衰処理
       └── グラフベースの関連メモリ検索
```

将来的にBackmanの専門エージェント分化を行う場合、グラフ管理は独立エージェントに切り出せる設計とする。

#### Frontmanへの情報伝達

グラフ情報のFrontmanへの受け渡しは、コンテキスト汚染防止の原則に従う。

```
Frontman → Backman: 「ユーザーが"上司"に言及した」
Backman → Frontman: サマリーのみ返却
  {
    entity: "上司",
    primary_emotion: {frustration: 0.7, trust: 0.3},
    active_tags: ["ストレス源", "1on1関連"],
    related_entities: ["Notionテンプレート", "部署異動"],
    confidence: 0.8
  }
```

詳細（個別の記憶内容、過去の会話コンテキスト等）が必要な場合、Frontmanが追加リクエストを行う（反復的取得パターン、最大3回ループ）。

#### event-driven search との関係

既存のevent-driven searchを置換するものではない。

- event-driven search：「何が起きたか」を時系列で検索
- 関係性グラフ：「何と何がどう繋がっているか」を構造で検索

両者は補完関係にある。Backmanは検索クエリの性質に応じて使い分ける。

## 4. ユースケース

### 4.1 関係性の変化を反映した応答

```
【過去の会話（2ヶ月前）】
ユーザー: 「上司と全然合わなくて辛い」
→ Edge(ユーザー, 上司) に Tag("ストレス源", strength:0.9) が生成

【現在の会話】
ユーザー: 「上司と最近いい感じで仕事できてる」
→ Tag("ストレス源") が減衰中（strength: 0.4）
→ 新規 Tag("協力者", strength:0.7) が生成
→ Amygdalaは「関係性が改善している」文脈を踏まえて応答
```

### 4.2 暗黙の関連を引き出す

```
ユーザー: 「仕事でストレス溜まってて」
→ Backmanがタグ"ストレス源"でグラフ検索
→ Edge(ユーザー, 上司) が Tag("ストレス源") を保持
→ Edge(ユーザー, 1on1) が Tag("未解決") を保持
→ Frontmanに「上司との関係性、1on1の未解決事項」がサマリーで渡る
→ ユーザーが明示しなくても文脈に合った応答が可能
```

### 4.3 話題の共起関係の活用

```
ユーザーが「Python」と「Amygdala」を頻繁に同じ文脈で言及
→ Edge(Python, Amygdala) の strength が上昇
→ ユーザーが「Pythonの新しいライブラリ見つけた」と言ったとき
→ Backmanが Amygdala への適用可能性も関連情報として返却
```

## 5. 技術的考慮事項

### 5.1 ストレージ

グラフデータは既存のメモリストレージと同一レイヤーに保持する。ノード数上限100、エッジ数上限2000（100ノード × 20エッジ）の規模であれば、JSONベースのフラットファイルでも性能上の問題は生じない。将来的なスケールが必要になった場合、SQLiteまたはグラフDB（Neo4j等）への移行パスを残す。

### 5.2 コンテキストバジェットへの影響

グラフ検索結果のFrontmanへの注入量を試算する。

- サマリー形式（上記3.4の例）：約100-200トークン/エンティティ
- 1回の会話で参照されるエンティティ：通常1-3個
- 追加コンテキスト消費：最大600トークン程度

200kコンテキストウィンドウに対して0.3%以下であり、実用上問題ない。1Mウィンドウではさらに余裕がある。

### 5.3 プライバシーとセーフティ

- グラフに保持される情報はユーザーが会話中に明示的に言及した内容のみから生成する
- 推論による関係性（例：「AさんとBさんは不仲」）は confidence を低く設定し、ユーザーに確認なく応答に反映しない
- ユーザーからの「忘れて」リクエストに対応するため、特定ノード/エッジ/タグの手動削除APIを提供する

## 6. 実装方針

### 6.1 段階的導入

| ステップ | 内容 | 依存関係 |
|----------|------|----------|
| Step 1 | ノードとエッジの基本データ構造を実装 | なし |
| Step 2 | 会話からのエンティティ抽出とノード自動生成 | Step 1 |
| Step 3 | タグの生成・昇格パイプライン | Step 2 |
| Step 4 | 減衰処理の実装 | Step 3 |
| Step 5 | Backmanへのグラフ検索統合 | Step 4 |
| Step 6 | Frontmanへのサマリー返却統合 | Step 5 |

各ステップは独立してテスト可能。Step 1-4は既存機能に影響を与えずに開発・テストできる。

### 6.2 既存機能への影響

- Frontman/Backman間のインターフェースに新しいメッセージタイプ（グラフクエリ/レスポンス）を追加
- 既存のevent-driven searchには変更を加えない
- soft-archive処理にグラフノード/エッジのアーカイブ処理を追加

## 7. 評価指標

| 指標 | 測定方法 | 目標 |
|------|----------|------|
| 関連性精度 | グラフから返された関連エンティティがユーザーの意図と合致する割合 | 80%以上 |
| コンテキスト消費 | グラフ情報注入による追加トークン消費量 | 500トークン以下/ターン |
| タグ精度 | 昇格したタグが実際の関係性を正しく表現している割合 | 70%以上 |
| 減衰妥当性 | 減衰したタグが「もう関連しない」関係性に対応している割合 | 75%以上 |

---

## 8. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 誤った関係性の推論 | ユーザーが不快に感じる応答 | confidence閾値による抑制、候補→昇格パイプライン |
| グラフ肥大化 | 検索速度低下、コンテキスト圧迫 | ノード/エッジ上限、自動soft-archive |
| 減衰パラメータの不適切さ | 重要な関係性が早期に消失 | 初期値は保守的に設定、運用データで調整 |
| 既存機能との競合 | event-driven searchとの重複検索 | 検索クエリの性質で明示的に経路を分離 |
| 重力モデルの再来（計算コスト爆発） | 性能劣化 | ホップ制限（最大2）、ノード上限（100）で物理的に制約 |

---

## 9. 将来の拡張可能性

- **Backmanの専門エージェント分化**：グラフ管理を独立エージェントに切り出し
- **意図推論レイヤー**：感情ベクトルの隣に「意図ベクトル」を追加し、エッジに載せる
- **グラフの可視化**：ユーザーが自分の関係性マップを確認・編集できるUI
- **マルチユーザー対応**：ユーザーごとに独立したグラフを管理（API key管理と連動）
- **グラフ間の構造比較**：異なるユーザーのグラフ構造パターンから、一般的な関係性テンプレートを抽出

## 付録：設計思想の出典

本提案は以下の設計思想を参考にしている（実装の直接的な依存はない）。

- **GSD（Get Shit Done）**: コンテキスト汚染防止、薄いオーケストレーターパターン、フェーズ分割の思想

- **Amygdala既存設計**: soft-archive原則、event-driven search、10軸感情ベクトル
